# 架構分析與模塊建議

> 本文件分析 AI Service Framework 目前的架構狀態，並提出可新增的模塊與優化方向。

---

## 一、現有架構總覽

### 核心層（core/）

| 模塊 | 類型 | 職責 | 成熟度 |
|------|------|------|--------|
| `_common` | 內部 | 基底類別、標準回應、例外處理、分頁 | ✅ 穩定 |

## 二、架構優勢

1. **解耦設計** — Event Bus 讓模組間零直接依賴
2. **一致性** — `StandardResponse`、`BaseModel`、`BaseViewSet` 確保 API 風格統一
3. **可插拔擴充** — AI Provider 與 Payment Gateway 使用 Registry + Decorator 註冊
4. **開發體驗** — `make dev` 一鍵啟動、dev_bootstrap 自動 migrate + superuser
5. **安全預設** — JWT-only 認證、API Key 加密儲存、敏感資料日誌過濾

## 三、建議新增模塊

> **分類結論**：以下五個模塊均為**外部核心模組**（`core/`），而非業務模組（`modules/`）。
> 判斷標準：「換一個完全不同的專案，還會原封不動地帶走這個模塊嗎？」— 五個都是 Yes。
> 詳細分析見各模塊的獨立設計文件。

### 1. `core/notifications` — 通知中心

> 📄 完整設計文件：[core-notifications.md](./core-notifications.md)

**問題**：目前只有 Email 發送（寫死在 `event_handlers.py`），無法支援多管道通知。

**為什麼放在 `core/` 而不是 `modules/`**：
通知是跨業務領域的基礎設施能力。派發引擎只知道「用哪個管道、發給誰」，不知道業務語意。
業務語意（通知內容、觸發條件）由呼叫者透過 Event Bus 注入。

**關鍵設計**：
- `BaseChannel` + `ChannelRegistry` — 和 `ProviderRegistry` / `GatewayRegistry` 同模式
- `Notification` + `NotificationDelivery` 分離（一則通知 → 多管道獨立追蹤）
- 使用者偏好管理 + 免打擾時段 + 安全通知強制管道

---

### 2. `core/file_storage` — 檔案儲存服務

> 📄 完整設計文件：[core-file-storage.md](./core-file-storage.md)

**問題**：目前頭像用 Django `FileField` 直存本機；無集中式檔案管理。

**為什麼放在 `core/` 而不是 `modules/`**：
和 `ai_providers` / `payments` 完全同模式 — 抽象「能力後端」。
它不知道「這個檔案是頭像還是訂單附件」，那是呼叫者的事。

**關鍵設計**：
- `BaseStorageBackend` + `StorageBackendRegistry` — dev 用 Local、prod 用 S3
- Presigned URL 直傳（大檔案不經過後端）
- `FileRecord` + `StorageQuota`（中繼資料 + 配額管理）
- 孤兒檔案偵測 + 暫存檔案過期清除

---

### 3. `core/audit_log` — 操作審計日誌

> 📄 完整設計文件：[core-audit-log.md](./core-audit-log.md)

**問題**：`_logger` 記錄的是技術日誌（request/response），缺少業務層面的「誰在什麼時候做了什麼」。

**為什麼放在 `core/` 而不是 `modules/`**：
審計是完全跨業務的橫切關注點。記錄結構和查詢方式和業務無關。
且 `AuditEntry` 是不可變的（不可修改、不可刪除），不繼承 `SoftDeleteMixin`。

**關鍵設計**：
- 透過 Event Bus `@subscribe("*")` 被動收集 — 業務模組不 import audit_log
- `AUDITABLE_EVENTS` 集中映射表 — 新增可審計事件只改一個字典
- 同步 / 非同步寫入策略分流（安全事件同步，高頻事件非同步）
- 敏感資料自動過濾

---

### 4. `core/rbac` — 角色權限管理

> 📄 完整設計文件：[core-rbac.md](./core-rbac.md)

**問題**：目前只有 `is_staff` / `is_superuser` 二元權限，無法做細粒度控制。

**為什麼放在 `core/` 而不是 `modules/`**：
RBAC 是認證授權系統的延伸（`core/auth` → 你是誰，`core/rbac` → 你能做什麼）。
業務模組只需宣告 `required_permissions`，權限檢查由框架自動完成。

**關鍵設計**：
- `Permission`（`{module}.{action}` 格式）+ `Role`（含繼承結構）
- `RBACPermission` DRF 類別 — 自動注入 `BaseViewSet`
- Redis 快取 + 主動失效（角色/權限變更時立即生效）
- `PermissionRegistry` — 各模組啟動時自動註冊自己的權限
- `sync_permissions` management command — 同步到資料庫

---

### 5. `core/api_keys` — API Key 管理

> 📄 完整設計文件：[core-api-keys.md](./core-api-keys.md)

**問題**：目前只有 JWT 認證，對於 server-to-server 或第三方整合場景不便。

**為什麼放在 `core/` 而不是 `modules/`**：
API Key 是認證機制之一，和 JWT 同層級。兩者在 DRF 的認證鏈條中共存，
共用相同的權限框架。這是純粹的框架能力。

**關鍵設計**：
- 資料庫只存 SHA-256 Hash（和密碼儲存同原理）
- `ask_` 前綴（AI Service Key）— 安全掃描友善
- Scope 和 RBAC Permission 使用相同命名格式，有效權限 = RBAC ∩ Scopes
- IP 白名單（CIDR 支援）
- Key 輪換（rotate）— 舊 Key 失效 + 新 Key 繼承設定
- Per-key 速率限制

## 四、核心層優化建議

### 4.1 `_event_bus` — 加入事件 Schema 驗證

**現況**：`publish_event()` 接受任意 dict payload，無型別檢查。

**建議**：引入 `EventSchema`（基於 Pydantic 或 dataclass），在 publish 時驗證 payload 結構。

```python
@dataclass
class UserRegisteredPayload:
    user_id: str
    email: str

publish_event("auth.user.registered", UserRegisteredPayload(user_id="...", email="..."))
```

**效果**：在 publish 端就能抓到 payload 錯誤，不用等 handler 端 KeyError。

---

### 4.2 `_common` — 新增 `BaseService` 基底類別

**現況**：各模組的 Service 各自為政（`AccountService`、`AIProviderService`、`PaymentService`），沒有統一基底。

**建議**：

```python
class BaseService:
    """業務邏輯服務基底。"""

    def __init__(self, user: User | None = None):
        self.user = user
        self.logger = get_logger(self.__class__.__module__)

    @transaction.atomic
    def execute(self, **kwargs):
        """子類覆寫此方法實現業務邏輯。"""
        raise NotImplementedError
```

**效果**：統一日誌注入、transaction 管理、異常處理模式。

---

### 4.3 `_task_queue` — 支援任務優先級

**現況**：所有 Celery 任務共用同一個 queue。

**建議**：在 `BaseTask` 加入 `queue` 屬性，搭配 Celery 的 `task_routes` 設定：

```python
class MyTask(BaseTask):
    task_type = "command"
    queue = "high_priority"   # 自動路由到 high_priority queue
```

**Celery 設定**：

```python
CELERY_TASK_ROUTES = {
    "*.high_priority.*": {"queue": "high_priority"},
}
```

**效果**：付款 webhook 等關鍵任務不會被大量 AI 生成任務卡住。

---

### 4.4 前端測試面板增強

**現況**：前端只能做 API 請求測試，無法看到系統整體狀態。

**建議新增面板**：

1. **系統狀態儀表板** — 整合 `/api/v1/system/health/` 的結果，顯示 DB / Redis / Celery 健康狀態
2. **Event Bus 視覺化** — 列出已註冊的事件 handler，方便 debug
3. **Task 監控** — 列出最近的 TaskProgress 紀錄，顯示進度與狀態

## 五、優先順序建議

| 優先級 | 模塊 / 優化 | 理由 |
|--------|------------|------|
| 🔴 高 | `core/notifications` | 取代散落的 `send_mail()`，統一通知出口 |
| 🔴 高 | `core/audit_log` | 業務追溯需求，越早建立越好 |
| 🟡 中 | `core/rbac` | 當系統有多角色使用者時必須 |
| 🟡 中 | `_event_bus` Schema 驗證 | 防止 payload 格式錯誤的隱性 bug |
| 🟡 中 | `core/api_keys` | server-to-server 整合需求，和 rbac 搭配使用 |
| 🟢 低 | `core/file_storage` | 等到有大量檔案需求時再建 |
| 🟢 低 | `_common` BaseService | 改善程度不大，可逐步遷移 |

---

## 六、架構依賴方向（不變原則）

```
業務模組（modules/）
    ↓ 可使用
外部核心模組（core/auth, accounts, ai_providers, payments,
             notifications, file_storage, audit_log, rbac, api_keys）
    ↓ 可使用
內部核心模組（core/_common, _logger, _event_bus, _task_queue）
    ↓ 可使用
Django / DRF / Celery 框架
```

**禁止方向**：
- ❌ 內部模組不可 import 外部模組
- ❌ 核心模組不可 import 業務模組
- ❌ 模組間不可直接 import（改用 Event Bus）
