## 一、核心層優化（Phase 1）

### 1.1 Event Bus Schema 驗證（`core/_event_bus`）

**檔案**：`schemas.py`（新增）、`bus.py`（修改）、`registry.py`（修改）、`__init__.py`（修改）

- 新增 `EventSchema` 基底 dataclass，定義事件 payload 結構
- 新增 `SchemaRegistry`，支援嚴格模式（raise ValueError）與寬鬆模式（log warning）
- 新增 `@register_schema` 裝飾器，便於宣告式註冊
- `publish_event()` 在發布時自動驗證 payload 是否符合 Schema
- `HandlerRegistry.get_handlers()` 支援 `"*"` 萬用字元訂閱（用於 audit_log）

### 1.2 BaseService 基底類別（`core/_common`）

**檔案**：`base_services.py`（新增）、`__init__.py`（修改）

- 新增 `BaseService` 基底類別，提供：
  - 自動 logger 注入（`self.logger`）
  - 使用者上下文（`self.user`、`self._require_user()`）
  - 事件發布快捷方法（`self._publish_event()`）
  - 系統上下文（`as_system()` class method）
- 新增 `@transactional` 裝飾器，自動包裹 `transaction.atomic`

### 1.3 Task Queue 優先級支援（`core/_task_queue`）

**檔案**：`constants.py`（新增）、`base_task.py`（修改）、`__init__.py`（修改）

- 新增 `TaskPriority` 常數類別（HIGH / DEFAULT / LOW）
- `BaseTask` 支援 `queue` 屬性，自動路由到對應 queue
- `settings/base.py` 新增 `CELERY_TASK_QUEUES` 配置

## 五、架構決策記錄

### 5.1 APIView vs BaseViewSet

**決策**：維持使用 `APIView`，不強制轉換為 `BaseViewSet`。

**理由**：整個既有 codebase（auth、accounts、ai_providers、payments）均使用 `APIView` 模式。
`BaseViewSet` / `BaseModelViewSet` 在 CRUD 密集的場景才有優勢，
大多數端點為操作型（如 revoke、rotate、read-all），`APIView` 更清晰。

### 5.2 BaseService 採用策略

**決策**：新模組建議使用 `BaseService`，但不強制全面改造。

**理由**：`BaseService` 為本次新增的基礎設施，`NotificationService` / `PreferenceService` 已採用。
其他模組使用 `@staticmethod` / `@classmethod` 模式也能正常運作，
未來可漸進式遷移。

### 5.3 AuditEntry 不可變設計

**決策**：`AuditEntry` 使用 `UUIDPrimaryKeyMixin` + `TimestampMixin`，不繼承 `BaseModel`。

**理由**：審計日誌必須不可變（immutable），軟刪除（SoftDeleteMixin）在審計場景中不適用。

### 5.4 APIKeyUsageLog 使用 BigAutoField

**決策**：`APIKeyUsageLog` 使用 `BigAutoField` 而非 UUID 作為主鍵。

**理由**：高流量日誌表需要自動遞增的整數 PK 以提升寫入效能。

### 5.5 delete_file() 配額回收邏輯

**決策**：在 `soft_delete()` 之前先 `save(update_fields=["status"])` 持久化狀態變更。

**理由**：`SoftDeleteMixin.soft_delete()` 僅儲存 `is_deleted` / `deleted_at` / `updated_at`，
不會持久化其他欄位變更。必須分兩步驟保存。
